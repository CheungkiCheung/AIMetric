# AIMetric 中文执行计划

## 1. 项目目标

AIMetric 目标是同构复现文章《AI出码率70%+的背后：高德团队如何实现AI研发效率的量化与优化》中的 AI 研发效能量化平台。

复现重点不是简单做一个 dashboard，而是复现文章里的核心链路：

```text
AI 编码工具 / MCP 工具
  -> 采集端
  -> 采集网关
  -> 事件存储
  -> Git / AI 归因
  -> 指标计算
  -> 个人 / 团队看板
```

文章同构分层：

- `采集平台层`：Cursor、CLI、IDE 插件等入口
- `数据采集层`：MCP 工具、采集 SDK、采集网关
- `平台能力层`：事件存储、归因、指标计算、规则体系、知识库查询
- `指标展示层`：个人出码视图、团队出码视图、会话分析、出码分析

## 2. 当前完成度

截至当前版本：

- `Phase 1 主链路 MVP`：约 `96%` 完成
- `全量文章同构系统`：约 `53%` 完成

当前远端 `main` 分支是稳定版本，已经推送到 GitHub。

## 3. Phase 1：主链路 MVP

目标：先跑通最小闭环，让系统可以从采集事件一路走到指标看板。

状态：已完成

内容：

- 初始化 `pnpm workspace`
- 建立 `apps/` 和 `packages/` 分层
- 配置 TypeScript、Vitest、基础 lint/typecheck
- 建立统一工程结构

核心目录：

- `apps/collector-gateway`
- `apps/metric-platform`
- `apps/mcp-server`
- `apps/dashboard`
- `packages/event-schema`
- `packages/metric-core`
- `packages/rule-engine`
- `packages/collector-sdk`
- `packages/git-attribution`

### 3.2 统一事件模型

状态：已完成

目标：

定义所有采集入口都能复用的事件协议，避免每个 IDE / CLI / MCP 工具各自上报一套格式。

已完成内容：

- `IngestionBatchSchema`
- `IngestionEventSchema`
- 事件批次校验
- 最小 `v1` schema

当前支持的典型事件：

- `session.started`
- `session.recorded`
- 后续可扩展 `beforeEditFile`、`afterEditFile`、`tab.accepted` 等事件

### 3.3 MCP 主链路工具

状态：已完成

目标：

复现文章中后期演进方向：通过 MCP 标准化采集替代强依赖单一 IDE 本地数据库的方案。

已完成工具：

- `beforeEditFile`
- `afterEditFile`
- `recordSession`

当前能力：

- 编辑前记录文件快照
- 编辑后生成最小 diff 证据
- 会话结束记录摘要

后续增强：

- 接入真实 MCP server runtime
- 增加文档查询 MCP
- 增加规则查询 MCP
- 增加工具调用审计

### 3.4 采集网关 collector-gateway

状态：已完成基础主链路

目标：

作为所有采集端的统一入口，负责接收、校验、转发采集事件。

已完成能力：

- `POST /ingestion`
- `GET /health`
- 按 `IngestionBatchSchema` 校验采集批次
- 将采集批次转发到 `metric-platform /events/import`

后续增强：

- 接入队列缓冲
- 增加重试机制
- 增加鉴权
- 增加限流
- 增加采集来源审计

### 3.5 指标平台 metric-platform

状态：已完成基础主链路，PostgreSQL 持久化、快照表和回算任务已完成

目标：

承接事件导入、归因证据、指标计算和对外查询 API。

已完成能力：

- `POST /events/import`
- `GET /metrics/personal`
- `GET /metrics/team`
- `GET /health`
- PostgreSQL 事件事实表
- PostgreSQL 指标快照表
- 个人快照计算
- 团队快照计算
- 手动指标回算接口
- 定时指标回算 scheduler
- 指标快照查询接口
- 基础 AI 出码率公式
- 采集事件进入平台后可更新指标快照

当前稳定版本说明：

- 已经打通采集网关到指标平台的数据链路
- 事件可以写入 PostgreSQL 事实表
- 个人/团队指标可以沉淀到 PostgreSQL 快照表
- 指标平台支持 `POST /metrics/recalculate`
- 指标平台支持 `GET /metrics/snapshots`
- 不同服务实例可以从数据库读取已写入事件
- 默认测试跳过真实数据库集成测试，避免本地未启动数据库时影响开发

后续增强：

- 缺陷率、返工率、质量指标联动

### 3.6 Git / AI 归因证据

状态：已完成基础版

目标：

复现文章中的关键统计口径：

```text
AI 出码率 = AI 采纳行数 / commit 代码总行数
```

已完成能力：

- `buildCommitEvidence`
- 匹配 AI 生成行与最终 commit 行
- 输出：
  - `acceptedAiLines`
  - `commitTotalLines`

当前是最小实现，先证明指标口径能跑通。

后续增强：

- 基于真实 git diff
- 支持文件路径、行号、commit sha
- 支持多 commit 聚合
- 支持重复行、移动行、格式化行处理
- 支持 AI 归因置信度

## 8. 下一步建议

建议优先级如下：

1. 增加规则中心与知识库 MCP
2. 增加员工接入插件原型
3. 增加准生产能力

最推荐下一步：

```text
规则中心与知识库 MCP
```

原因：

- 当前主链路、PostgreSQL 事实表、Dashboard 筛选、自动刷新、快照表和回算任务已经跑通
- 下一步应补文章中的规则分层与知识库查询能力
- 规则中心和知识库 MCP 是员工简单接入、规则注入、研发规范落地的核心基础

## 9. 当前风险

- 当前 dashboard 已支持筛选和自动刷新，但仍缺少趋势图和会话明细
- MCP server 目前是工具函数层，还不是完整 MCP runtime 服务
- Git 归因仍是基础行匹配，后续需要接真实 diff
- 规则体系还只是基础包，没有接入知识库
