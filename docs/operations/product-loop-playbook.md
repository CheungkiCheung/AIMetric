# AIMetric 企业 AI 工具度量产品闭环

本文档定义 AIMetric 的企业级产品闭环。它不是单纯展示“AI 出了多少代码”，而是帮助企业回答：哪些 AI 工具真的被使用，哪些场景产生了有效交付，效率提升有没有带来质量代价，以及下一步应该推广、修复还是治理。

## 1. 产品定位

AIMetric 是面向提效管理者、技术管理者和平台管理员的 AI 工具度量平台。

核心边界：

- 度量对象是 `AI 工具、研发场景、交付链路和质量风险`
- 管理目标是 `推广决策、工具治理、试点复盘和持续优化`
- 员工侧目标是 `轻量接入、低打扰、可排障、可解释`
- 平台侧目标是 `多工具采集、统一指标语义、可扩展数据接入`

明确不做：

- 不把员工个人产出排名作为默认产品目标
- 不用单一出码率代表全部 AI 价值
- 不在证据链不足时直接承诺精确 ROI
- 不要求员工为了度量改变正常开发习惯

## 2. 目标用户与核心场景

### 提效管理者

要回答的问题：

- 公司投入的 AI 工具有没有真正用起来
- 哪些团队、工具、场景值得继续推广
- 哪些工具接入了但没有形成有效产出
- 哪些效率收益伴随明显质量代价

典型动作：

- 看 AI 工具度量驾驶舱
- 对比工具渗透、产出转化、交付影响和风险成本
- 形成推广、培训、暂停、补采集或深入分析的管理动作

### 技术管理者

要回答的问题：

- AI 是否进入本团队真实需求和 PR
- AI 触达需求是否更快流向交付
- Review、CI、缺陷、发布失败是否出现风险偏移
- 团队应优先改善工具使用方式还是工程流程

典型动作：

- 查看团队 / 项目 / 需求 / PR / 质量归因
- 解释指标口径是否符合本团队工程语义
- 输出团队改进建议和风险修复计划

### 平台管理员

要回答的问题：

- 员工端采集是否稳定
- 不同 AI 工具是否能统一接入
- 权限、身份映射、审计和采集健康是否可控
- 后续扩展 GitLab、缺陷系统、IDE、Agent 工具是否有统一入口

典型动作：

- 配置 collector、token、viewer scope、identity mapping
- 监控 `/ready`、`/metrics`、采集失败和 outbox 积压
- 排查采集链路、数据导入和权限边界

### 员工

要回答的问题：

- 我需要安装什么
- 会不会影响编码体验
- 网络不可用时会不会阻塞
- 采集了什么，能否排障

典型动作：

- 执行 onboarding 命令
- 设置 collector token 环境变量
- 用 `doctor / status / flush` 自助检查
- 正常使用 Cursor、CLI Agent、Codex、Claude Code、VS Code 或 JetBrains

## 3. 一条完整产品闭环

```text
AI 工具资产登记
  -> 员工轻量接入
  -> 多源采集与导入
  -> 采集健康监控
  -> 指标语义层计算
  -> 工具 / 团队 / 场景多维分析
  -> 质量与交付归因
  -> 管理动作
  -> 试点复盘与下一轮推广
```

每个环节的产品目标：

- `AI 工具资产登记`：知道企业有哪些 AI 提效工具，哪些已接入，哪些接入就绪
- `员工轻量接入`：把安装和配置压到最少，不打断日常开发
- `多源采集与导入`：既采 AI 工具使用，也接需求、PR、CI、发布、事故、缺陷
- `采集健康监控`：先确认数据可信，再解释指标
- `指标语义层计算`：统一口径，避免每个页面各算各的
- `多维分析`：按工具、团队、项目、场景、时间看趋势和差异
- `质量与交付归因`：把 AI 参与和交付结果、缺陷风险串起来
- `管理动作`：把看板结论转为推广、培训、治理、暂停、补采集
- `试点复盘`：形成下一轮范围、工具和指标口径

## 4. 数据与指标分层

### 自动采集层

适合通过员工端、MCP、SDK、IDE/CLI Adapter 自动采集：

- AI 会话数、轮次、使用时长、活跃人数
- 工具来源、项目、成员、仓库、时间窗口
- Tab 接受行数、AI 生成代码行数、编辑证据
- MCP 工具调用成功率、失败率、平均耗时
- collector delivery mode、队列深度、DLQ、转发失败

### 工程链路导入层

适合通过 API、Webhook、定时同步或 CSV 导入：

- 需求：Jira、TAPD、禅道等
- PR / MR：GitHub、GitLab
- CI：GitHub Actions、GitLab CI、Jenkins
- 发布：部署平台、流水线平台
- 事故：On-call、监控告警、ITSM
- 缺陷：缺陷系统、测试平台、线上反馈

### 语义计算层

当前优先沉淀的企业级指标：

- 使用渗透：AI 工具活跃人数、会话数、工具覆盖率、场景覆盖率
- 有效产出：AI 出码率、Tab 接受行数、AI 触达 PR 占比
- 交付效率：需求到 PR 时间、PR 周转时间、Lead Time、关键需求周期
- 质量风险：Review 退回率、CI 通过率、缺陷率、逃逸缺陷率、变更失败率、回滚率
- 归因分析：AI 参与需求缺陷率、AI 触达 PR 逃逸缺陷率、发布失败关联缺陷数、事故关联缺陷数
- 业务价值：当前先保留轻量口径，优先看 critical requirement cycle time，不急于做复杂财务模型

### 人因与经营扩展层

适合后续用问卷、访谈、Agent/RAG 或运营系统补齐：

- 不使用 AI 工具原因
- 满意度、信任度、认知负荷
- 新人上手时间
- 单位需求成本
- 提效预估与 ROI

## 5. 产品页面地图

### 第一入口：AI 工具度量驾驶舱

服务对象：提效管理者。

页面应该优先回答：

- 哪些 AI 工具已接入，哪些接入就绪
- 每个工具的渗透、产出、交付、质量表现如何
- 本周期最值得推广、修复、治理的工具是什么
- 哪些数据还不可信，需要平台先补采集

### 第二入口：团队与交付分析

服务对象：技术管理者。

页面应该优先回答：

- AI 触达需求和 PR 的比例
- AI 参与链路与非 AI 链路的交付差异
- 质量风险是否集中在特定团队、项目或工具
- 哪些工程实践需要改进

### 第三入口：采集与治理中心

服务对象：平台管理员。

页面应该优先回答：

- 哪些工具采集正常，哪些工具有失败或延迟
- 哪些团队 outbox 积压、token 失效或身份映射异常
- viewer scope、collector identity、审计记录是否满足企业要求

### 第四入口：员工接入中心

服务对象：员工和平台支持人员。

页面应该优先回答：

- 当前工具如何安装
- 需要设置哪些环境变量
- 如何检查采集状态
- 网络失败时如何 flush

## 6. 管理动作模型

看板必须能沉淀成动作，而不是只停留在观察。

建议动作类型：

- `扩大推广`：工具渗透高、有效产出高、质量风险可控
- `补充培训`：使用渗透低，但已使用人群效果好
- `修复采集`：指标异常来自采集失败、身份映射缺失或数据导入不完整
- `质量治理`：交付更快但缺陷、Review、CI、发布失败上升
- `暂停推广`：工具使用增加但没有有效产出，且带来明显质量风险
- `深入分析`：需要 Agent/RAG 读取更多上下文、代码变更和缺陷说明

## 7. 一个月企业试点节奏

### 第 0 周：准备

- 选定 `1-2` 个团队和 `10-30` 名研发
- 确认至少两类 AI 工具
- 配置 collector、metric-platform、dashboard
- 明确数据边界和隐私说明

### 第 1 周：接入

- 完成员工 onboarding
- 验证 `doctor / status / flush`
- 监控采集健康和 outbox
- 只判断“采得到、不中断”

### 第 2 周：校准

- 接入需求、PR、CI、发布、缺陷数据
- 校准团队、项目、成员映射
- 确认核心指标口径
- 暂不急于做经营 ROI

### 第 3 周：分析

- 看工具渗透、有效产出、交付效率、质量风险
- 对比 AI 触达和非 AI 触达链路
- 形成第一批管理动作
- 标记需要补采集或修口径的地方

### 第 4 周：决策

- 决定扩大、暂停、补培训、补采集或进入深度分析
- 输出试点复盘
- 明确下一轮工具、团队、指标和治理动作

## 8. 企业级护栏

为了避免 AI 度量系统被误用，产品默认遵守这些护栏：

- 先看团队、工具和场景，再看个人排障信号
- 默认不把个人 AI 使用数据直接作为绩效评价
- 指标必须展示口径、数据来源和采集健康状态
- 采集端不保存真实 token，只引用环境变量名
- 网络失败不阻塞员工开发，事件进入 outbox 后异步 flush
- 敏感字段、源码片段和提示词内容应按企业策略脱敏或最小化采集
- 管理动作需要结合业务上下文，不用单一指标自动判定团队好坏

## 9. 下一阶段优先级

在 Agent / RAG 智能分析之前，下一阶段更应该优先补齐：

1. 工具 x 团队 x 场景的趋势钻取
2. 指标口径说明和数据来源可解释
3. 采集异常到管理动作的闭环
4. 员工接入中心的安装包 / 脚本化体验
5. GitLab、缺陷系统、CI、发布平台的更多真实连接器
6. 代码审查、安全审查、接口契约和端到端测试收敛

当这些基础稳定后，再接入 Agent / RAG，用它解释“为什么某个工具有效或无效”，而不是让 Agent 代替基础指标平台。
