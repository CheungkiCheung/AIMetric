# AIMetric 提效工具体验与效果总览 Walkthrough

本文档用于试点演示和联调，帮助提效部门负责人按一条固定路径理解 AIMetric 的价值。演示主线不是“页面有哪些”，也不是“今天要推进什么”，而是“哪些 AI 提效工具真的被员工持续使用，哪些只是覆盖或试用，为什么不好用，以及这些工具是否产生了可解释的提效效果”。

## 1. 演示前准备

先确保：

- `collector-gateway` 已启动
- `metric-platform` 已启动
- `dashboard` 已启动
- 已导入一批演示数据

导入演示数据：

```bash
corepack pnpm demo:seed
```

只看导入计划，不真正写入：

```bash
corepack pnpm demo:seed:dry
```

先做联调检查：

```bash
corepack pnpm demo:check
```

如果检查失败，优先确认：

- `collector-gateway` 是否运行在 `127.0.0.1:3000`
- `metric-platform` 是否运行在 `127.0.0.1:3001`
- 当前 `projectKey` 是否与导入数据一致

## 2. 推荐演示顺序

建议按下面顺序讲：

1. `提效工具体验与效果总览`
   先说明 AIMetric 的第一入口是面向提效部门负责人的产品首页。核心关注覆盖、持续使用、有效会话、体验反馈和提效效果，不把覆盖率当成唯一目标。
2. `工具资产与接入状态`
   展示已接入、接入就绪、观测中工具，以及工具类别、健康状态和当前聚焦工具。这里强调工具分类：AI-IDE、SDD、Co-Claw/OpenClaw、自研提效工具。
3. `工具体验矩阵`
   展示每类工具的覆盖率、持续使用人数比例、有效会话、使用频率和体验状态。
4. `不用 / 低频 / 不好用原因`
   展示覆盖但未持续使用、试用后流失、低频使用和负反馈原因。
5. `提效效果`
   展示人效提升比例、人力节省比例、SDD 需求交付占比、AI 代码生成率和 AI 生成代码采纳率。
6. `AI-IDE 效果`
   展示 AI-IDE 使用人数比例、AI 代码生成行数、Tab 接受行数、AI 生成代码采纳率、AI 代码生成率。
7. `SDD 规约驱动开发`
   展示 SDD 使用人数比例、SDD 开发需求数、SDD 需求交付占比、AI-IDE + SDD 需求覆盖率、规约模板和流程入口反馈。
8. `Co-Claw / OpenClaw`
   展示使用人数比例、使用频率、有效会话、一次完成情况和结果采纳情况。不要把参与批次、结对记录、批次完成状态作为 Co-Claw 本体指标。
9. `代码度量`
   展示总代码变更行数、AI 代码生成行数、人均代码产出和代码产出（行/人天）。
10. `团队与场景矩阵`
   对比团队需求总数、AI-IDE 开发需求数、SDD 开发需求数、各团队 AI-IDE 覆盖率、各团队 SDD 覆盖率和不用原因。
11. `试点运营信息`
   如存在结对编程或专项试点，再展示批次目标、时间窗口和完成状态。它是运营背景，不应被混入 Co-Claw/OpenClaw 的工具本体指标。
12. `质量护栏`
   再展示 Lead Time、PR 周转、Review、CI、缺陷、发布失败、事故关联，避免“局部更快、整体更差”。
13. `改进动作板`
   把结论落到改工具体验、改流程入口、补充样例、补充培训、修复采集、质量治理、暂停或扩大试点。
14. `组织治理概览`
   先说明这不是单一看板，而是有组织、团队、项目、成员治理边界的平台。
15. `权限治理配置`
   说明技术管理者和平台管理员看到的范围可以不同。
16. `采集健康运营`
   说明平台先保证“采得到、采得稳”，再谈指标。
17. `个人出码视图 / 团队出码视图`
   展示 AI 使用渗透和基础有效产出。
18. `需求交付概览 / PR 交付概览`
   展示 AI 是否真正进入需求与交付流程。
19. `CI 质量概览 / 发布质量概览 / 事故风险概览 / 缺陷风险概览`
   展示速度和质量是否平衡。
20. `缺陷归因分析`
   重点讲：
   - `AI 参与需求缺陷率`
   - `AI 触达 PR 逃逸缺陷率`
   - `发布失败关联缺陷数`
   - `事故关联缺陷数`
21. `统一指标计算管线`
   说明平台现在不是散点指标，而是统一语义层。

## 3. 建议重点讲的业务问题

演示时不要只讲“系统做了什么”，更建议围绕问题讲：

- 哪些工具真的被持续使用
- 哪些工具只是覆盖、安装或试用
- 覆盖但未持续使用的原因是什么
- 员工觉得不好用的具体原因是什么
- AI 代码生成行数、AI 代码生成率、AI 生成代码采纳率是否提升
- SDD 是否作为规约驱动开发流程进入需求，而不是被当成 Agent 评估
- 团队 AI-IDE 覆盖率、SDD 覆盖率和需求覆盖率差异在哪里
- Co-Claw/OpenClaw 的使用覆盖、使用频率、有效会话和一次完成情况如何
- 不用 AI-IDE、不用 SDD 的原因是什么
- 人力节省、人效提升、SDD 需求交付占比是否接近目标
- AI 是否进入需求和 PR，而不只是停留在聊天
- 速度提升有没有带来 Review、CI、缺陷、事故风险
- 哪些风险可以归因到需求、PR、发布或事故链路
- 哪些指标已经足够做试点，哪些经营指标还不该过早做重
- 下一步应该改工具、改入口、补样例、补培训、修复采集、质量治理，还是调整试点范围

## 5. 演示结束时可以给出的结论

当前这版最适合给出的结论是：

```text
这套系统已经可以支撑企业 AI 研发提效试点，
重点价值不只是“看 AI 出码率”或“看覆盖率”，
而是把 AI-IDE、SDD、Co-Claw/OpenClaw、自研提效工具的真实使用、体验反馈、代码产出、需求交付、质量护栏和改进动作连成一条产品闭环。
```

不建议在这个阶段过度承诺：

- 精准 ROI
- 精确人力成本节省
- 全组织统一经营结论

更好的说法是：

```text
先把趋势、对比、归因和治理做扎实，再进入 Agent / RAG 智能分析阶段。
```
