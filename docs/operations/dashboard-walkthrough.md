# AIMetric AI 提效经营驾驶舱 Walkthrough

本文档用于试点演示和联调，帮助提效管理者按一条固定路径理解 AIMetric 的价值。演示主线不是“页面有哪些”，而是“AI-IDE、SDD、结对编程和内部 AI 工具是否铺开，是否形成代码产出与采纳，是否覆盖真实需求，是否接近人效提升、人力节省和 SDD 需求交付占比目标”。

## 1. 演示前准备

先确保：

- `collector-gateway` 已启动
- `metric-platform` 已启动
- `dashboard` 已启动
- 已导入一批演示数据

导入演示数据：

```bash
corepack pnpm demo:seed
```

只看导入计划，不真正写入：

```bash
corepack pnpm demo:seed:dry
```

先做联调检查：

```bash
corepack pnpm demo:check
```

如果检查失败，优先确认：

- `collector-gateway` 是否运行在 `127.0.0.1:3000`
- `metric-platform` 是否运行在 `127.0.0.1:3001`
- 当前 `projectKey` 是否与导入数据一致

## 2. 推荐演示顺序

建议按下面顺序讲：

1. `AI 提效经营驾驶舱`
   先说明 AIMetric 的第一入口是面向提效管理者的经营驾驶舱，核心关注 AI-IDE、SDD、结对编程、Co-Claw/OpenClaw 和内部 AI 工具的覆盖、产出、采纳、批次推进和终极人效目标。
2. `工具资产与接入状态`
   展示已接入、接入就绪、观测中工具，以及工具类别、健康状态和当前聚焦工具。
3. `终极目标`
   展示人效提升比例、人力节省比例、SDD 需求交付占比和 AI 代码生成率。
4. `AI-IDE + SDD 使用`
   展示 AI-IDE 使用人数比例、SDD 使用人数比例、AI-IDE + SDD 需求覆盖率、AI 生成代码采纳率。
5. `代码度量`
   展示总代码变更行数、AI 代码生成行数、人均代码产出和代码产出（行/人天）。
6. `团队覆盖矩阵`
   对比团队需求总数、AI-IDE 开发需求数、SDD 开发需求数、各团队 AI-IDE 覆盖率、各团队 SDD 覆盖率和不用原因。
7. `结对编程批次`
   展示批次推进目标、批次时间窗口、批次完成状态和参与人数增长。
8. `质量护栏`
   再展示 Lead Time、PR 周转、Review、CI、缺陷、发布失败、事故关联，避免“局部更快、整体更差”。
9. `管理动作板`
   把结论落到扩大推广、补充培训、修复采集、质量治理、暂停推广或深入分析。
10. `组织治理概览`
   先说明这不是单一看板，而是有组织、团队、项目、成员治理边界的平台。
11. `权限治理配置`
   说明技术管理者和平台管理员看到的范围可以不同。
12. `采集健康运营`
   说明平台先保证“采得到、采得稳”，再谈指标。
13. `个人出码视图 / 团队出码视图`
   展示 AI 使用渗透和基础有效产出。
14. `需求交付概览 / PR 交付概览`
   展示 AI 是否真正进入需求与交付流程。
15. `CI 质量概览 / 发布质量概览 / 事故风险概览 / 缺陷风险概览`
   展示速度和质量是否平衡。
16. `缺陷归因分析`
   重点讲：
   - `AI 参与需求缺陷率`
   - `AI 触达 PR 逃逸缺陷率`
   - `发布失败关联缺陷数`
   - `事故关联缺陷数`
17. `统一指标计算管线`
   说明平台现在不是散点指标，而是统一语义层。

## 3. 建议重点讲的业务问题

演示时不要只讲“系统做了什么”，更建议围绕问题讲：

- AI 有没有真正被用起来
- AI-IDE 和 SDD 有没有在目标团队铺开
- AI 代码生成行数、AI 代码生成率、AI 生成代码采纳率是否提升
- 团队 AI-IDE 覆盖率、SDD 覆盖率和需求覆盖率差异在哪里
- 结对编程批次是否达成人数增长和完成状态目标
- 不用 AI-IDE、不用 SDD 的原因是什么
- 人力节省、人效提升、SDD 需求交付占比是否接近目标
- AI 是否进入需求和 PR，而不只是停留在聊天
- 速度提升有没有带来 Review、CI、缺陷、事故风险
- 哪些风险可以归因到需求、PR、发布或事故链路
- 哪些指标已经足够做试点，哪些经营指标还不该过早做重
- 下一步管理动作是扩大、培训、修复采集、质量治理还是暂停推广

## 4. 本轮最适合演示的指标

建议重点挑这些：

- `AI 出码率`
- `AI-IDE 使用人数比例`
- `SDD 使用人数比例`
- `AI-IDE + SDD 需求覆盖率`
- `AI 生成代码采纳率`
- `总代码变更行数`
- `AI 代码生成行数`
- `代码产出（行/人天）`
- `结对编程参与人数`
- `Co-Claw / OpenClaw 使用人数比例`
- `AI 触达需求占比`
- `AI 触达 PR 占比`
- `lead_time_ai_vs_non_ai`
- `critical_requirement_cycle_time`
- `pr_cycle_time`
- `review_rejection_rate`
- `ci_pass_rate`
- `change_failure_rate`
- `rollback_rate`
- `defect_rate`
- `escaped_defect_rate`
- `AI 参与需求缺陷率`
- `AI 触达 PR 逃逸缺陷率`

## 5. 演示结束时可以给出的结论

当前这版最适合给出的结论是：

```text
这套系统已经可以支撑企业 AI 研发提效试点，
重点价值不只是“看 AI 出码率”，
而是把 AI-IDE、SDD、结对编程、代码产出、AI 采纳、团队覆盖、批次推进、质量护栏和管理动作连成一条产品闭环。
```

不建议在这个阶段过度承诺：

- 精准 ROI
- 精确人力成本节省
- 全组织统一经营结论

更好的说法是：

```text
先把趋势、对比、归因和治理做扎实，再进入 Agent / RAG 智能分析阶段。
```
