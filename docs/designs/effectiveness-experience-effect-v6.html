/ 工具体验

AI-IDE

核心是开发者是否持续把它放进编码流程。看使用人数比例、活跃天数、有效会话、AI 生成代码采纳率、Tab 接受行数和 AI 代码生成率。

  • 当前判断覆盖不是主要问题，持续使用和采纳质量更关键。
  • 下一步问题低频员工是不会用、场景不适配，还是生成质量不稳定。

SDD

SDD 是规约驱动开发流程，不是 Agent。它应围绕需求规约、开发过程和交付结果观测，而不是按代码行数硬套。

  • 应该看SDD 使用人数比例、SDD 开发需求数、AI-IDE + SDD 需求覆盖率、SDD 需求交付占比。
  • 体验诊断入口是否清楚、规约模板是否可用、需求类型是否适合 SDD。

Co-Claw / OpenClaw

它更接近轻量 AI 辅助工具或小龙虾式工具体验，不应该承接 AI-IDE 的批次运营指标。

  • 应该看使用人数比例、使用频率、有效会话、一次完成情况、结果采纳。
  • 不应该看参与批次、结对记录、批次完成状态不作为 Co-Claw 本体指标。

自研提效工具

按工具服务的场景定义结果：评审工具看问题发现和采纳，测试工具看用例生成和缺陷拦截，自动化工具看耗时减少和成功率。

  • 应该看覆盖、频率、成功率、耗时、结果采纳、节省工时估算。
  • 体验诊断失败原因、入口路径、等待时间、输出是否能直接进入工作流。

结果指标

看效果，不看待办
  • 人力节省比例节省工时 / 原始投入工时，必须带估算口径和置信度。
  • SDD 需求交付占比SDD 参与并完成交付需求 / 完成需求。
  • AI 生成代码采纳率被采纳 AI 代码 / AI 建议代码，用来判断生成质量。
  • 人均代码产出代码变更行数 / 参与开发人数。

效果解释

避免单指标误判

AI 代码生成率高不等于真实提效，需要同时看采纳率、返工、需求交付和员工反馈。

SDD 交付占比低不一定代表工具无效，也可能是适用需求少、入口不清或规约模板没有统一。

Co-Claw 覆盖低先看频率和一次完成情况，再判断是否需要改入口、提示或适用场景。

/ 指标口径

使用指标

  • 覆盖率目标人群中具备使用条件或有使用记录的人数比例。
  • 活跃天数周期内发生有效工具事件的天数。
  • 有效会话数产生采纳、生成、交付、完成或明确结果的会话。
  • 持续使用活跃天数 + 有效会话数共同判断。

代码指标

  • 总代码变更行数Git / PR diff 采集。
  • AI 代码生成行数AI-IDE 生成并进入工作区的代码。
  • AI 代码生成率AI 代码生成行数 / 总代码变更行数。
  • 代码产出代码变更行数 / 人天。

效果指标

  • 人力节省比例节省工时 / 原始投入工时。
  • SDD 需求交付占比SDD 参与并完成交付需求 / 完成需求。
  • 人效提升比例单位人力完成需求或产出提升。
  • 一次完成情况Co-Claw/OpenClaw 输出一次可用或少量修改可用的比例。
